# 使用說明

## 本知識庫用途

本知識庫用於協助 GPT 將統包需求書轉譯成 HTML 需求圖譜。若使用者同時提供審查意見書，應依 `09_審查意見整合規則.md` 擷取弱電相關問題與回覆，並整合進同一份 HTML。預設另依 `10_MD旁支輸出規則.md` 產出 `to_xmind.md` 與 `todo_handoff.md`，作為可編輯樹狀圖與弱電案件 AI 交接檔。

預設只處理：
1. 專案基本資訊
2. 智慧建築標章規定
3. 弱電系統設備工程設計原則
4. 機電設備規格需求說明表

預設不產生：
- 完整 RFI 總表
- 完整 CAD 任務表
- 未追溯、不可匯入後續流程的自由 JSON
- M1.5 結構化資料包，除非使用者明確要求
- BIM 任務
- 正式智慧建築得分判斷
- 法規最終符合性判斷
- 合約責任判斷

## 輸出目標

GPT 應將統包需求書轉譯成：
- HTML Dashboard
- 分析結果摘要總覽
- 智慧建築對應矩陣
- 弱電系統總覽表
- 弱電需求樹狀圖 / 心智圖
- 依系統展開的需求細節
- 機電設備規格總覽表
- 依系統展開的設備規格細節
- 弱電相關審查意見問題 / 回覆對照
- 需人工確認清單
- 基本待辦與釐清清單
- 可編輯弱電樹狀圖 `to_xmind.md`
- 弱電案件 AI 交接報告 `todo_handoff.md`

## 基本限制

- 智慧建築只能寫「可能對應」
- 不可寫「已符合」
- 不可寫「可得分」
- 不可判斷正式通過
- 不可自行決定設備數量、位置、線路路徑、品牌型號
- 模組一主要供人員閱讀、查核與釐清目標，不作為後續系統匯入的主要資料源
- `to_xmind.md` 是人工可編修樹狀圖，不是正式責任分工表
- `todo_handoff.md` 是弱電案件交接報告，不是 M1.5 結構化資料包
- 不處理 BIM

## 來源追溯原則

- 每筆被引用的需求都應保留來源頁碼。
- 每筆被引用的需求都應保留短原文摘錄。
- 若需求書沒有頁碼，應標示「來源頁碼待確認」。
- 若原文過長，只擷取會影響設計、設備、圖說、送審或施工的短摘錄。

## 預設工作順序

1. 擷取專案基本資訊。
2. 擷取智慧建築標章相關要求。
3. 擷取弱電系統設備工程設計原則。
4. 擷取機電設備規格需求說明表中與弱電、中央監控、能源管理、智慧建築或系統整合有關的內容。
5. 若有審查意見書，擷取弱電相關問題與回覆；非弱電相關意見不輸出明細。
6. 依系統分類並轉譯為工程可讀語言。
7. 建立智慧建築「可能對應」矩陣。
8. 建立基本待辦與釐清清單。
9. 輸出摘要總覽、HTML Dashboard、矩陣、XMIND 模式樹狀圖、總覽表、審查意見頁籤與折疊細節。
10. 依同一輪擷取結果輸出 `to_xmind.md` 與 `todo_handoff.md`；不得為了 MD 旁支檔重新擴張範圍到非弱電系統。
