# M1.5 選用資料包輸出規則

## 目的

M1.5 是選用資料包，不是模組一的預設主線。只有使用者明確要求系統匯入、批次比對、API 串接或後續自動化時，才輸出本資料包。

模組一的預設主交付仍是給人看的 `demand_map.html`。M1.5 資料包只是把 HTML 需求圖譜中的部分內容整理成可追溯、可審核、可匯入後續系統的輔助資料。

## 輸出原則

- 主交付仍是 `demand_map.html`。
- 不預設輸出 `m1_5_data_package.zip`。
- 使用者明確要求 M1.5 或系統匯入時，才另外輸出 `m1_5_data_package.zip`。
- 不建議把完整 JSON 內嵌在 HTML；HTML 是給人看的，系統資料應獨立成檔。
- 所有 M1.5 欄位都必須能追溯到 HTML 可讀內容或來源文件。
- 不可把 M1.5 資料包寫成正式 CAD 任務、正式 RFI、正式報價或智慧建築審查結論。

## 檔案清單

| 檔案 | 用途 |
|---|---|
| requirements.csv | 後續 M2 CAD、M4 報價與人工審核共用的需求主表 |
| requirements.json | 與 CSV 同內容，供程式或後續 GPT 直接讀取 |
| manual_review.csv | 需人工確認清單，作為審核與釋疑前置資料 |
| equipment_candidates.csv | 從需求與規格中抽出的設備候選，供設備表、品項 mapping 與圖塊任務使用 |
| handoff_report.md | 說明本次輸出完整度、缺口、風險與可否進入 M2 / M4 |

## requirements 欄位

| 欄位 | 說明 |
|---|---|
| requirement_id | 穩定需求編號，建議格式 REQ-{{SYSTEM_CODE}}-001 |
| project_id | 專案代碼；若未提供，填 PROJECT-ID-待確認 |
| source_doc | 來源文件名稱 |
| source_page | 來源頁碼；不明時填 來源頁碼待確認 |
| source_section | 來源章節、表格或圖說位置 |
| original_excerpt | 短原文摘錄，不貼長段原文 |
| system_code | 弱電系統代碼，例如 CCTV、LAN、BMS |
| system_name | 系統中文名稱 |
| floor_or_area | 樓層或區域；不明時填 未明確提供 |
| space_or_room | 空間或房間；不明時填 未明確提供 |
| translated_task | 轉譯後工程任務 |
| drawing_or_document_need | 可能需要的圖說、設備表、架構圖、點表或送審文件 |
| smart_building_possible_indicator | 智慧建築可能對應指標，不得寫正式符合 |
| review_status | draft、needs_review、reviewed 三選一；預設 draft 或 needs_review |
| review_reason | 需要人工確認的原因；無則填 無 |
| detail_anchor | 對應 HTML details id，例如 #system-cctv |

## manual_review 欄位

| 欄位 | 說明 |
|---|---|
| review_id | 穩定確認編號，建議格式 REV-{{SYSTEM_CODE}}-001 |
| requirement_id | 對應需求編號；若為全案問題，填 PROJECT-GENERAL |
| source_page | 來源頁碼 |
| related_section | 對應 HTML 區塊或來源章節 |
| issue | 需確認事項 |
| why_it_matters | 為什麼需要確認 |
| suggested_owner | 建議確認對象 |
| priority | 高、中、低 |
| detail_anchor | 對應 HTML details id 或人工確認區塊 id |

## equipment_candidates 欄位

| 欄位 | 說明 |
|---|---|
| equipment_candidate_id | 穩定設備候選編號，建議格式 EQ-{{SYSTEM_CODE}}-001 |
| requirement_id | 對應需求編號 |
| system_code | 系統代碼 |
| equipment_name | 設備名稱或設備群 |
| spec_focus | 規格重點 |
| interface_or_protocol | 介面或協定；不明時填 需人工確認 |
| integration_need | 整合需求 |
| quantity_status | 明確、未明確、需人工確認 |
| location_status | 明確、未明確、需人工確認 |
| item_master_hint | 可能對應的預算品項提示；不可當成正式報價品項 |
| review_status | draft、needs_review、reviewed 三選一 |

## HTML 內嵌規則

若使用者要求同時在 HTML 中提示資料包狀態，可在頁尾聲明前新增簡短 `M1.5 選用資料包摘要` 區塊，至少呈現：

- 需求主表筆數
- 人工確認筆數
- 設備候選筆數
- 可否進入 M2 CAD 圖塊任務
- 可否進入 M4 報價品項 mapping
- 阻擋原因或主要缺口

只有使用者明確要求單檔 HTML 同時供系統抽取時，才在 `</main>` 前內嵌：

```html
<script type="application/json" id="m1-5-requirements-data">
[]
</script>
<script type="application/json" id="m1-5-manual-review-data">
[]
</script>
<script type="application/json" id="m1-5-equipment-candidates-data">
[]
</script>
```

JSON 必須是有效 JSON 陣列，不可使用 Markdown 表格、註解或尾逗號。

## 進入後續模組的判斷

### 可進入 M2 CAD 前置

- 主要系統已有 `system_code`。
- 需求已可轉成圖說或設備表任務。
- 樓層、空間、設備位置若不明，已列入 `manual_review.csv`。
- 每個系統 details id 穩定且可被連結。

### 可進入 M4 報價前置

- 設備候選已可對應到設備類型或材料類型。
- 數量不明的設備已標示 `quantity_status`。
- 可能需要線材、管材、工資或測試費的項目已在 `drawing_or_document_need` 或 `item_master_hint` 標示。
- 未對應品項不可隱藏，必須保留為異常或待確認。

## 禁止事項

- 不可用 M1.5 直接產生正式報價。
- 不可把 `item_master_hint` 當成正式品項主檔。
- 不可把 `equipment_candidates.csv` 當成正式設備表。
- 不可因為資料不足而刪除需求；應保留並標示 `needs_review`。
- 不可輸出無法追溯到來源頁碼、來源章節或 HTML details 的統計數字。
