# 需求樹狀圖層級規則

## 用途

此檔案用於規範 GPT 產生 HTML 需求圖譜時的樹狀圖層級。  
樹狀圖不是系統摘要，而是要把統包需求書中的弱電需求轉成可閱讀的需求階層。

v0.2 起，需求樹是模組一的核心資料結構。GPT 必須先產生「唯一需求樹」，再由同一份樹派生：

- HTML 的 `需求樹核心`
- `to_xmind.md` 的主節點
- `詳細資訊查詢` 的對應 anchor

不得讓 HTML 需求樹、`to_xmind.md` 與詳細資訊查詢各自使用不同分類或不同節點名稱。

## 樹狀圖最低層級要求

HTML 中的弱電需求樹狀圖至少要有四層：

第一層：需求大類  
第二層：系統分類  
第三層：子系統 / 空間 / 設備群  
第四層：具體需求節點  

必要時可加入第五層：

第五層：來源頁碼 / 需人工確認 / 智慧建築關聯 / 圖說需求 / 介面協定

第五層只作為 HTML 標籤列或詳細資訊查詢依據，不得輸出成 `to_xmind.md` 主節點。

## 第一層：需求大類

可使用：

- 弱電需求
- 特殊空間需求
- 智慧建築標章規定
- 機電設備規格需求說明表
- 跨系統整合需求
- 需人工確認事項

- 中央監控系統
- 監視系統
- 門禁管制系統
- 停車管理
- 視聽音響
- 廣播系統
- 防盜保全系統
- 緊急求救系統
- 公共資訊多媒體系統
- 對講系統
- 空調 / IAQ
- 能源管理
- 電力監控
- 給排水監控
- 消防連動
- 電梯連動
- 照明控制
- 公眾通信涵蓋

## 第四層：具體需求節點

具體需求節點必須是可讀的需求，不可只寫模糊摘要。

範例：
- 各單位辦公室及服務空間設置電話插座
- 禮廳、小禮堂、停柩區設置電話與網路插座
- 屋頂層預留數位電視天線接管路
- 各空間設置電視插座
- 中央監控需納入電力、給排水、電梯、消防、照明、空調、門禁、緊急求救與監視系統
- 重要出入口、屋頂、房間、機房設門禁
- 公共空間依場地需求設攝影機
- 停車系統需考量車牌辨識、收費、車位偵測、充電車位管理
- 特殊空間需依樓層與空間整理影音、顯示、廣播、門禁、監視與求救需求

## 第五層：補充標籤

每個具體需求節點可附加：

- 來源頁碼
- 需人工確認
- 智慧建築可能關聯
- 可能需要圖說
- 介面 / 協定
- 整合需求

## 不可接受的樹狀圖

不可只輸出：

弱電系統設備工程設計原則
- 通訊與網路基礎
- 中央監控與能源
- 安全防災與保全
- 停車與資訊顯示服務

這種層級太粗，不符合需求。

也不可輸出成橫向層級分欄，例如：

| 第一層 | 第二層 | 第三層 | 第四層 |
| --- | --- | --- | --- |
| 弱電需求 | 電話系統 | PBX / 電話插座 | 各辦公室依需求設電話點 |
| 弱電需求 | 資訊網路 | MDF / IDF | 各層設資訊機櫃 |

這種表格或橫向卡片欄位看似層級完整，但使用者不容易追蹤主從關係，也不符合心智圖用途。

HTML 需求樹應採 `demand_map (4).html` 的「一個系統一張可折疊卡片」：

- 外層 `details.tree-category`：需求大類。
- 內層 `details.tree-system`：系統分類。
- 第一層 `ul.tree-list` 縮排：子系統 / 空間 / 設備群。
- 第二層縮排：具體需求摘要。
- 第三層縮排或標籤：來源頁碼、智慧建築可能關聯、需人工確認、責任界面待確認。
- 不要把來源頁碼、需確認、智慧建築關聯做成獨立欄位。
- 每個具體需求節點應有 `查看詳情` 或 anchor，連到詳細資訊查詢區。
- `to_xmind.md` 只複製第一層到第四層的主節點文字，不複製來源頁碼、狀態標籤或詳情連結。

## 可接受的樹狀圖

應輸出類似：

弱電需求
- 電話系統
  - 內外線系統 PBX
  - 各單位辦公室及服務空間電話插座
  - 電梯預留對講機
  - 1F 入口各層門廳預留公共電話插座
- 資訊系統
  - MDF / IDF
  - 網路機櫃
  - 交換器
  - Cat.6 佈線
  - 各禮廳、小禮堂、停柩區電話與網路插座

  - 電力系統
    - 多功能電表
