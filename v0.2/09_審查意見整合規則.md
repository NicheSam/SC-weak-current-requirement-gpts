# 審查意見整合規則

## 目的

當使用者同時提供統包需求書與審查意見書時，GPT 應將審查意見作為統需書的補充、修正與追蹤層，整合到 HTML 需求圖譜中。

審查意見不是新的原始需求書，也不可被 GPT 自行判定為正式設計結論。若審查回覆與統需書內容衝突、補充或未能追溯，必須列入審查意見區與待辦釐清清單。

## 適用輸入

可能輸入型態包括：
- 統包需求書 PDF
- 審查意見書 PDF
- 審查會議紀錄
- 各階段報告審查意見
- 表格式審查意見與回覆
- 掃描 PDF 或 OCR 文字
- Word / Excel 轉出的文字內容

審查意見格式可能不一致，不要求固定欄位名稱。GPT 應以語意辨識「問題」與「回覆」。

## 弱電相關篩選

只擷取弱電、中央監控、能源管理、智慧建築弱電佐證或跨系統介面相關的審查意見。

應納入的常見關鍵範圍：
- 弱電、低壓、通訊、電話、網路、LAN、Wi-Fi、光纖、MDF、IDF
- 監視、CCTV、攝影機、錄影、影像管理
- 門禁、讀卡、出入口、管制、訪客、權限
- 停車管理、車牌辨識、車位偵測、停車誘導、繳費
- 中央監控、BMS、IBMS、圖控、I/O、DDC、PLC、控制器
- 能源管理、EMS、電力監視、需量控制、分項用電
- 對講、廣播、緊急求救、防盜保全、電子布告欄、資訊顯示、視聽音響
- IAQ、CO2、PM2.5、TVOC、溫濕度、照度、水質等環境感測
- 消防、電梯、照明、給排水、空調等系統與 BMS / EMS / 門禁 / CCTV / 智慧建築有介面或連動時
- 智慧建築手冊中需要弱電、監控、感測、資料庫、系統整合或佐證文件的項目

不納入的內容：
- 純土建、結構、景觀、裝修、一般材料、法規程序或行政文件，且未涉及弱電或智慧建築弱電佐證
- 純消防、電梯、空調、給排水問題，且沒有監控、連動、資料、介面或智慧建築關聯
- 純工期、契約、經費、採購程序，且未影響弱電系統範圍或設備需求

非弱電相關意見不需逐筆輸出；可在 Dashboard 或審查意見區簡短標示「已排除非弱電相關意見」，不列明細。

## 問題與回覆辨識

以下都可視為「問題」欄位：
- 問題
- 審查意見
- 委員意見
- 審查項目
- 缺失
- 建議
- 待修正事項
- 說明不足
- 意見內容

以下都可視為「回覆」欄位：
- 回覆
- 回覆情形
- 答覆
- 處理情形
- 辦理情形
- 修正說明
- 設計單位回覆
- 主辦機關回覆
- 已修正內容
- 回應說明

若只有問題、沒有回覆，仍需保留弱電相關問題，並標示「尚未見明確回覆」。

若只有回覆、無法找到對應問題，仍可保留，但需標示「原審查問題待確認」。

## 每筆審查意見輸出欄位

- 編號或原表序號
- 來源文件名稱
- 來源頁碼或章節
- 審查委員或提出單位
- 審查階段或日期
- 問題 / 審查意見短摘錄
- 回覆 / 處理情形短摘錄
- 關聯弱電系統
- 與統需書的關係：補充、修正、衝突、追蹤、僅提醒
- 對 HTML 需求圖譜的影響
- 是否需要人工確認
- 建議確認對象
- 對應系統 details anchor

## 整合到 HTML 的方式

HTML 必須在「詳細資訊查詢」中新增審查意見區。

審查意見頁籤應包含：
- 弱電相關審查意見總覽
- 問題 / 回覆對照表
- 關聯系統
- 對統需書需求的影響
- 需追蹤或需人工確認事項

若審查意見明確影響某一弱電系統，該系統 details 區塊可補一段「審查意見補充」，但不得取代原統需書來源。

若審查意見與統需書矛盾，必須：
1. 在「詳細資訊查詢」的審查意見區標示矛盾。
2. 在「詳細資訊查詢」的待辦釐清區加入待確認項。
3. 不自行判定以哪一份文件為準。

## 禁止事項

- 不輸出非弱電相關審查意見明細。
- 不把審查回覆當成正式設計結論。
- 不自行判定審查意見已完全解除疑義。
- 不因回覆寫「已修正」就刪除待確認事項；若缺頁碼、圖號或修正位置，仍需標示需人工確認。
- 不將審查意見整理成正式 RFI 或合約責任判斷。
